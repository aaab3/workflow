> 本手册详细解释每个模块的工作原理、配置细节和实际案例。即使你从未接触过编程，看完也能灵活使用。

| 模块 | 图标 | 一句话说明 | 类比 |
|------|------|-----------|------|
| LLM 对话 | 🤖 | 把文字发给 AI，AI 回复你 | 像微信问一个超级聪明的朋友 |
| 读取文件 | 🔌 | 从电脑里读一个文件的内容 | 像打开一本书，把文字抄出来 |

| 延时等待 | 🔀 | 暂停一会儿再继续 | 像设了个闹钟 |

## 🤖 LLM 对话（llm-chat）

### 它是什么？

这个模块让你在工作流里调用 AI 大语言模型（比如 GPT-4o、Claude）。你给它一段文字，它会像 ChatGPT 一样回复你。

### 工作原理

```
你的问题（文字） → [发送到 AI 服务器] → AI 思考 → 回复（文字）
```

底层实现：模块把你的消息打包成一个 HTTP 请求，发送到 OpenAI 兼容的 API 接口（默认走 OpenClaw 代理 localhost:18789），等待 AI 返回结果。

### 输入端口

| 端口名 | 类型 | 说明 |
|--------|------|------|
| 用户消息 | string (绿色) | 你想问 AI 的内容。可以是一个问题、一段需要处理的文本、一个指令 |

### 输出端口

| 端口名 | 类型 | 说明 |
|--------|------|------|
| 回复 | string (绿色) | AI 的回答文本 |
| 用量 | object (紫色) | Token 消耗统计：prompt_tokens（输入花了多少）、completion_tokens（输出花了多少）、total_tokens（总计） |
| 结束原因 | string (绿色) | 为什么停止：stop=正常结束，length=达到最大长度被截断 |

### 配置项详解

**模型（model）** — 必填
- `gpt-4o`：最聪明，适合复杂任务（翻译、分析、写作），但最贵最慢
- `gpt-4o-mini`：性价比最高，速度快，适合简单任务
- `gpt-3.5-turbo`：最便宜最快，但能力有限
- `claude-3-opus`：Anthropic 的最强模型
- `claude-3-sonnet`：平衡性能和成本
- `claude-3-haiku`：最快最便宜

**系统提示词（systemPrompt）** — 可选但强烈建议填
告诉 AI "你是谁"。这决定了 AI 的行为方式。

例子：
- "你是一个专业的中英翻译官，只输出翻译结果，不要解释"
- "你是一个代码审查专家，指出代码中的问题并给出修改建议"
- "你是一个文章摘要助手，用3句话总结以下内容"
- "你是一个 JSON 格式化工具，把输入的文本转换为结构化 JSON"

**温度（temperature）** — 默认 0.7
控制 AI 回答的"创造力"：
- 0 = 每次回答完全一样（适合翻译、提取信息等需要确定性的任务）
- 0.3 = 基本一致，偶尔有小变化（推荐用于大多数任务）
- 0.7 = 有一定随机性（适合写作、头脑风暴）
- 1.5-2.0 = 非常随机（适合创意写作，但可能胡说八道）

**最大 Token 数（maxTokens）** — 默认 2048
限制 AI 回复的最大长度。1 个 token 约等于 0.75 个英文单词或 0.5 个中文字。
- 短回答（一两句话）：设 256
- 普通回答：设 1024-2048
- 长文章/代码：设 4096-8192

**API 地址（baseUrl）** — 默认 http://127.0.0.1:18789/v1
一般不需要改。如果你没有运行 OpenClaw，可以改成：
- OpenAI 官方：https://api.openai.com/v1
- 其他兼容服务的地址

### 实际案例

**案例 1：翻译文件**
- 系统提示词："你是专业翻译，将以下英文翻译为中文，保持原文格式"
- 上游连一个「读取文件」节点，把文件内容传到"用户消息"端口
- 下游连一个「写入文件」节点，把"回复"端口的翻译结果保存

**案例 2：代码审查**
- 系统提示词："审查以下代码，列出问题和改进建议，用 Markdown 格式输出"
- 输入：代码文件内容
- 输出：审查报告

**案例 3：数据提取**
- 系统提示词："从以下文本中提取所有人名和日期，以 JSON 数组格式输出"
- 温度设为 0（需要确定性输出）
- 输出模式：下游用 JavaScript 节点 JSON.parse 解析

## 🔌 读取文件（io-file-read）

### 它是什么？

从你电脑上读取一个文件，把文件里的文字内容取出来，传给下游节点。

### 工作原理

```
文件路径 → [打开文件，读取全部内容] → 文字内容 + 文件大小
```

底层实现：调用 Node.js 的 `fs.readFile` 函数，按指定编码读取文件的全部内容到内存中。

### 输入端口

无。路径在配置里填写。

### 输出端口

| 端口名 | 类型 | 说明 |
|--------|------|------|
| 内容 | string (绿色) | 文件里的全部文字 |
| 大小 | number (蓝色) | 文件大小（字节数）。1KB=1024字节，1MB=1048576字节 |

### 配置项详解

**路径（path）** — 必填
文件在电脑上的位置。支持：
- 绝对路径：`C:/Users/你的用户名/Documents/笔记.txt`
- 相对路径：`./data/input.txt`（相对于项目根目录）
- 表达式：`{{input.filePath}}`（从工作流输入参数获取）

注意：Windows 路径用 `/` 或 `\\` 都行，推荐用 `/`。

**编码（encoding）** — 默认 utf-8
- `utf-8`：万能编码，中文英文日文都能读（99% 的情况用这个）
- `ascii`：纯英文文件
- `base64`：读取二进制文件（图片等）的 base64 编码
- `binary`：原始二进制数据

### 常见问题

- **报错"文件不存在"**：检查路径是否正确，注意大小写
- **读出来是乱码**：编码选错了，试试 utf-8
- **文件太大怎么办**：目前会全部读入内存，建议不要读超过 50MB 的文件

### 实际案例

**案例：读取配置文件并解析**
1. 读取文件节点：路径填 `./config.json`
2. 连接到 JavaScript 节点：`return JSON.parse(inputs.content);`
3. 现在你就有了一个 JSON 对象可以用了

## 🔌 写入文件（io-file-write）

### 它是什么？

把数据保存到电脑上的一个文件里。如果文件不存在会自动创建，如果存在默认会覆盖。

### 工作原理

```
内容 + 路径 → [创建/打开文件，写入内容] → 成功/失败
```

底层实现：调用 Node.js 的 `fs.writeFile`，如果父目录不存在会自动创建（`mkdir -p`）。

### 输入端口

| 端口名 | 类型 | 说明 |
|--------|------|------|
| 内容 | string (绿色) | 要写入的文字。如果不连线，会使用配置里的 content 字段 |

### 输出端口

| 端口名 | 类型 | 说明 |
|--------|------|------|
| 成功 | boolean (黄色) | true=写入成功，false=失败 |
| 路径 | string (绿色) | 实际写入的文件路径 |

### 配置项详解

**路径（path）** — 必填
保存到哪里。支持表达式，比如 `./output/{{input.fileName}}.txt`

**内容（content）** — 可选
如果没有连线到"内容"输入端口，就用这里填的内容。优先级：输入端口 > 配置项。

**编码（encoding）** — 默认 utf-8

**追加模式（append）** — 默认关闭
- 关闭：每次写入会覆盖文件原有内容
- 开启：在文件末尾追加，不会删除原有内容（适合写日志）

**自动创建目录（createDirs）** — 默认开启
如果路径中的文件夹不存在，自动创建。比如写入 `./a/b/c/file.txt`，会自动创建 a/b/c 三层目录。

### 实际案例

**案例：AI 翻译后保存**
1. 读取文件 → LLM 翻译 → 写入文件
2. 写入文件配置：路径 `./output/translated.md`

**案例：追加日志**
1. 每次执行把结果追加到日志文件
2. 配置：路径 `./logs/workflow.log`，追加模式=开启

## 🔌 HTTP 请求（io-http-request）

### 它是什么？

像浏览器一样访问一个网址，获取数据或发送数据。可以调用任何 Web API。

### 工作原理

```
URL + 方法 + 数据 → [发送 HTTP 请求到服务器] → 服务器响应数据
```

底层实现：使用 Node.js 内置的 `fetch` API 发送请求。

### 输入端口

| 端口名 | 类型 | 说明 |
|--------|------|------|
| 请求体 | any (灰色) | POST/PUT 请求时发送的数据。GET 请求不需要 |

### 输出端口

| 端口名 | 类型 | 说明 |
|--------|------|------|
| 响应 | any (灰色) | 服务器返回的数据（通常是 JSON 对象或文本） |
| 状态码 | number (蓝色) | HTTP 状态码：200=成功，404=找不到，500=服务器错误 |
| 响应头 | object (紫色) | 服务器返回的头信息 |

### 配置项详解

**URL** — 必填
要访问的网址。比如：
- `https://api.github.com/users/octocat` — 获取 GitHub 用户信息
- `https://httpbin.org/post` — 测试 POST 请求

**方法（method）** — 默认 GET
- `GET`：获取数据（最常用，像浏览器打开网页）
- `POST`：发送数据（像提交表单）
- `PUT`：更新数据
- `DELETE`：删除数据

**请求头（headers）** — 可选
有些 API 需要认证，在这里填：
```json
{ "Authorization": "Bearer your-api-key" }
```

**超时（timeout）** — 默认 30000ms（30秒）
如果服务器超过这个时间没响应，就报错。

**响应解析（responseType）** — 默认 json
- `json`：自动把响应解析为 JSON 对象（最常用）
- `text`：原样返回文本
- `arraybuffer`：二进制数据

### 实际案例

**案例 1：获取天气数据**
- URL：`https://api.weatherapi.com/v1/current.json?key=xxx&q=Beijing`
- 方法：GET
- 输出的"响应"就是天气 JSON 数据

**案例 2：发送消息到飞书/钉钉机器人**
- URL：你的 Webhook 地址
- 方法：POST
- 输入端口连一个 JavaScript 节点，构造消息体：
  ```js
  return { msgtype: "text", text: { content: inputs.message } };
  ```

## 🔌 终端/Agent（io-terminal）

### 它是什么？

这是最强大的模块之一。它可以运行任何终端命令，包括调用外部 AI Agent（Claude CLI、Aider、自定义脚本等）。

### 工作原理

```
输入数据 → [启动终端进程] → 通过 stdin 传入数据 → 进程处理 → 从 stdout 收集输出
```

底层实现：使用 Node.js 的 `child_process.spawn` 启动子进程，通过管道（pipe）进行 stdin/stdout 通信。

### 为什么这个模块很重要？

市面上几乎所有 AI Agent 都有命令行接口。通过这个模块，你可以把任何 Agent 编排进工作流：
- Claude Code：`claude -p "你的问题"`
- Aider：`aider --message "修复这个 bug"`
- Open Interpreter：`interpreter --message "分析这个数据"`
- 任何 Python/Node.js 脚本

### 输入端口

| 端口名 | 类型 | 说明 |
|--------|------|------|
| 输入内容 | string (绿色) | 通过 stdin 传给进程的数据。比如把文件内容传给 AI Agent 处理 |

### 输出端口

| 端口名 | 类型 | 说明 |
|--------|------|------|
| 输出 | string (绿色) | 进程的 stdout 输出（命令的执行结果） |
| 错误 | string (绿色) | 进程的 stderr 输出（错误信息） |
| 退出码 | number (蓝色) | 0=成功，非0=失败 |

### 配置项详解

**命令（command）** — 必填
要执行的终端命令。支持完整的 shell 语法（管道、重定向等）。

例子：
- `echo "hello"` — 最简单的测试
- `claude -p "翻译为中文"` — 调用 Claude
- `python process.py` — 运行 Python 脚本
- `cat file.txt | grep "error"` — shell 管道

**输入模式（stdinMode）** — 默认 text
- `none`：不传任何数据给进程（进程不需要输入时用）
- `text`：把上游数据当纯文本传入 stdin
- `json`：把上游数据 JSON 序列化后传入 stdin（适合传结构化数据给脚本）

**输出模式（outputMode）** — 默认 text
- `text`：原样输出 stdout 内容
- `json`：尝试把 stdout 解析为 JSON 对象（如果你的脚本输出 JSON）
- `lastLine`：只取最后一行（适合命令输出很多但你只要最后的结果）

**超时（timeout）** — 默认 120000ms（2分钟）
AI Agent 可能需要较长时间思考，建议设大一些（5-10分钟）。

**工作目录（cwd）** — 可选
命令在哪个目录下执行。比如你的项目在 `C:/projects/myapp`，就填这个路径。

**Shell 模式（shell）** — 默认开启
开启后支持管道（|）、重定向（>）、通配符（*）等 shell 语法。

### 实际案例

**案例 1：用 Claude 处理文本**
```
command: claude -p "总结以下内容为3个要点"
stdinMode: text
```
上游连一个「读取文件」节点 → 文件内容通过 stdin 传给 Claude → Claude 的回答从 stdout 输出

**案例 2：运行 Python 数据处理脚本**
```
command: python analyze.py
stdinMode: json
outputMode: json
```
上游数据 JSON 序列化后传给 Python 脚本，脚本处理后输出 JSON 结果

**案例 3：串联多个 Agent**
```
读取代码文件 → [Claude: 找出bug] → [Aider: 修复bug] → 写入修复后的文件
```

## ⚡ JavaScript 代码（code-javascript）

### 它是什么？

一个万能的数据处理节点。你可以写任何 JavaScript 代码来转换、计算、过滤数据。

### 工作原理

```
输入数据 → [在安全沙箱中执行你的代码] → return 的结果作为输出
```

底层实现：代码在 Node.js 的 vm 沙箱中执行（隔离环境，不能访问文件系统和网络）。

### 输入端口

| 端口名 | 类型 | 说明 |
|--------|------|------|
| 输入数据 | any (灰色) | 上游传来的任何数据，通过 `inputs` 对象访问 |

### 输出端口

| 端口名 | 类型 | 说明 |
|--------|------|------|
| 结果 | any (灰色) | 你 `return` 的内容 |

### 怎么写代码？

**规则 1：通过 `inputs` 获取上游数据**
```javascript
// 如果上游是「读取文件」节点，连到了 "content" 端口
const text = inputs.content;
```

**规则 2：通过 `return` 输出结果**
```javascript
// 返回一个对象，每个字段对应一个输出
return { result: "处理后的数据", count: 42 };
```

**规则 3：可以用 console.log 调试**
```javascript
console.log("当前数据:", inputs);  // 会出现在执行日志里
```

### 可用的全局对象

| 对象 | 说明 |
|------|------|
| `inputs` | 上游传来的数据 |
| `config` | 节点的配置参数 |
| `console` | 日志输出（log/warn/error） |
| `JSON` | JSON 解析和序列化 |
| `Math` | 数学函数 |
| `Date` | 日期时间 |
| `Array/Object/String/Number` | 基础类型方法 |
| `RegExp` | 正则表达式 |
| `Map/Set` | 集合类型 |

**不可用**（安全限制）：`require`、`fetch`、`fs`、`process`、`setTimeout`

### 配置项详解

**代码（code）** — 必填
你的 JavaScript 代码。

**超时（timeout）** — 默认 30000ms
代码最多执行多久。如果死循环会被强制终止。

### 代码模板

**模板 1：文本转大写**
```javascript
return { content: inputs.content.toUpperCase() };
```

**模板 2：按行分割并过滤空行**
```javascript
const lines = inputs.content.split('\n').filter(l => l.trim());
return { lines, count: lines.length };
```

**模板 3：JSON 解析**
```javascript
const data = JSON.parse(inputs.content);
return { name: data.name, age: data.age };
```

**模板 4：字符串模板拼接**
```javascript
const greeting = `你好 ${inputs.name}，今天是 ${new Date().toLocaleDateString()}`;
return { message: greeting };
```

**模板 5：数组处理**
```javascript
const items = inputs.data;
const total = items.reduce((sum, item) => sum + item.price, 0);
const expensive = items.filter(item => item.price > 100);
return { total, expensive, count: items.length };
```

## 🔀 条件分支（flow-condition）

### 它是什么？

根据一个条件判断，决定数据走哪条路。就像马路上的岔路口：满足条件走"真"出口，不满足走"假"出口。

### 工作原理

```
数据进来 → [判断条件是否成立] → 成立：从"真"出口出去
                              → 不成立：从"假"出口出去
```

### 输入端口

| 端口名 | 类型 | 说明 |
|--------|------|------|
| 判断值 | any (灰色) | 要判断的数据 |

### 输出端口

| 端口名 | 类型 | 说明 |
|--------|------|------|
| 真 | any (灰色) | 条件成立时，原始数据从这里出去 |
| 假 | any (灰色) | 条件不成立时，原始数据从这里出去 |

### 配置项详解

**运算符（operator）** — 默认 truthy

| 运算符 | 含义 | 例子 |
|--------|------|------|
| `truthy` | 数据存在且不为空 | "hello" → 真，"" → 假，0 → 假 |
| `falsy` | 数据为空或不存在 | "" → 真，null → 真，"hello" → 假 |
| `==` | 等于比较值 | 判断值="yes"，比较值="yes" → 真 |
| `!=` | 不等于比较值 | 判断值="no"，比较值="yes" → 真 |
| `>` | 大于 | 判断值=10，比较值=5 → 真 |
| `<` | 小于 | 判断值=3，比较值=5 → 真 |
| `>=` | 大于等于 | |
| `<=` | 小于等于 | |
| `contains` | 包含某段文字 | 判断值="hello world"，比较值="world" → 真 |
| `startsWith` | 以某段文字开头 | 判断值="hello"，比较值="he" → 真 |
| `endsWith` | 以某段文字结尾 | 判断值="hello"，比较值="lo" → 真 |
| `empty` | 为空 | ""→真，null→真，[]→真 |
| `notEmpty` | 不为空 | "hello"→真，[1,2]→真 |

**比较值（compareValue）** — 部分运算符需要
和什么比较。比如运算符选"contains"，比较值填"error"，意思是"判断输入的文字是否包含 error"。

### 实际案例

**案例 1：检查 AI 回复是否包含"是"**
- 上游：LLM 节点的"回复"端口 → 条件分支的"判断值"
- 运算符：contains
- 比较值：是
- "真"出口 → 继续处理
- "假"出口 → 结束或走其他路

**案例 2：检查 HTTP 状态码**
- 上游：HTTP 请求的"状态码"端口 → 条件分支的"判断值"
- 运算符：==

## 🔀 延时等待（flow-delay）

### 它是什么？

让工作流暂停一段时间再继续执行下一个节点。数据会原样传过去，不做任何修改。

### 工作原理

```
数据进来 → [等待 N 毫秒] → 数据原样出去
```

### 为什么需要它？

1. **API 限流**：很多 API 限制每秒只能调用几次，加延时避免被封
2. **等待外部系统**：比如等文件生成完成
3. **调试**：放慢执行速度，方便观察每一步

### 配置项

**等待时间（duration）** — 默认 1000ms
- 1000 = 1 秒
- 5000 = 5 秒
- 60000 = 1 分钟

### 实际案例

**案例：批量调用 API 时加延时**
```
循环节点 → 延时等待(2000ms) → HTTP 请求 → 下一次循环
```
每次请求之间等 2 秒，避免触发 API 的频率限制。
